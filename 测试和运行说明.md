# OpenFang 战术闭环 — 测试与运行说明

本文档说明如何在本地构建、启动 OpenFang daemon，并通过 **Dashboard Tactical 页** 或 **HTTP API** 验证「快环态势 + 慢环认知规划 + 人工授权」闭环。

---

## 1. 前置条件

| 项目 | 说明 |
|------|------|
| Rust 工具链 | 已安装 `cargo`（建议 stable） |
| 配置文件 | 默认路径：`%USERPROFILE%\.openfang\config.toml` |
| API 地址 | 默认 `http://127.0.0.1:4200` |
| CLI 二进制 | `target\release\openfang.exe` 或 `target\debug\openfang.exe` |
| 仿真（可选） | 连接 ArkSIM 时需先启动仿真；示例想定见项目 `protobuf/` 与 `docs/plan-arksim.md` |

> **注意**：启动 daemon 的命令是 `openfang start`，不是 `daemon`。

## 构建和启动
cargo build --release -p openfang-cli
#开发模式
cargo build -p openfang-cli
target\release\openfang.exe start
curl http://127.0.0.1:4200/api/platform/pending

## 2. 配置

### 2.1 使用战术示例配置

仓库根目录提供了示例：

```text
openfang.tactical.toml.example
```

复制并合并到用户配置（按需修改 LLM、ArkSIM 地址、干预规则）：

```powershell
Copy-Item openfang.tactical.toml.example $env:USERPROFILE\.openfang\config.toml
```

**必须检查的配置项：**

```toml
[platform]
mode = "simulation"          # 或 hardware / hybrid

[platform.adapters.primary]
type = "arksim"              # 或 dds
host = "127.0.0.1"
port = 5000

[platform.planning]
enabled = true               # 开启慢环
interval_secs = 10.0
llm_refine = false           # 首次联调建议先关 LLM，只用规则基线
# llm_base_url / llm_api_key 填入后再设 llm_refine = true
```

### 2.2 开启 Mission 方案人工审批（Confirm）

在 `[platform.intervention]` 中增加规则，使慢环 `mission_approval` 检查点进入 Pending：

```toml
[[platform.intervention.rules]]
stage = ["mission_approval"]
platform_ids = []            # 空 = 任意平台
command_classes = []
sources = []
mode = "confirm"             # 或 quorum
quorum = 1
window_s = 300.0
```

武器打击仍建议使用既有 `authorized_target` 规则（目标授权一次、后续自动闭环）：

```toml
[[platform.intervention.rules]]
stage = ["weapon_release"]
platform_ids = ["usv-01"]
command_classes = ["weapon"]
sources = ["llm", "workflow"]
mode = "authorized_target"
quorum = 1
window_s = 30.0
```

## 4. 启动与停止

### 4.1 停止已有 daemon

```powershell
Get-Process openfang -ErrorAction SilentlyContinue | Stop-Process -Force
Start-Sleep -Seconds 3
```

### 4.2 启动

```powershell
# 可选：若 default_model 使用 Groq 等云端 provider
$env:GROQ_API_KEY = "<your-key>"

Start-Process -FilePath "target\release\openfang.exe" -ArgumentList "start" -WindowStyle Hidden
Start-Sleep -Seconds 8
curl.exe -s http://127.0.0.1:4200/api/health
```

期望返回：`{"status":"ok","version":"..."}`

### 4.3 停止

```powershell
Get-Process openfang -ErrorAction SilentlyContinue | Stop-Process -Force
```

## 5. Dashboard — Tactical 作战态势台

1. 浏览器打开：`http://127.0.0.1:4200/#tactical`
2. 左侧导航：**System → Tactical**
3. 页面每 3 秒轮询 `/api/platform/pending`，展示五块内容：
   - **意图注入**：objective / priority_tracks / issued_by
   - **快环实时态势**：snapshot（平台、航迹、武器）
   - **慢环认知**：assessment（威胁、打击机会）
   - **规划链路**：mission → tasks → tactics
   - **人工干预台**：Pending 方案、目标授权、engagement 签署

### 5.1 典型联调流程

1. 确认页面右上角 **Slow loop ON**（`[platform.planning] enabled = true`）
2. 在「意图注入」填写 objective，点击 **Inject Intent**
3. 等待慢环周期（默认 10s），观察 **慢环认知 / 规划链路**
4. 若出现 **Plan held for approval**，点击 **Approve Plan**（按方案指纹批准）
5. 下一慢环周期应下发 standing plan；快环开始执行战术 intent
6. 对打击机会点击 **Authorize**；对 pending engagement 点击 **Sign & Launch**

> `planning.enabled = false` 或 platform 未启用时，页面会显示降级提示，snapshot / cognitive_report 可能为空。

## 6. HTTP API 联调

以下示例适用于 **PowerShell**（JSON 建议写入临时文件，避免转义问题）。

### 6.1 读取战术全景

```powershell
curl.exe -s http://127.0.0.1:4200/api/platform/pending
```

主要字段：

| 字段 | 含义 |
|------|------|
| `planning_enabled` | 慢环是否启用 |
| `snapshot` | 快环最新态势 |
| `cognitive_report` | 慢环最新报告（含 `pending_approval_id`） |
| `intent_queue` | 未消费的 commander intent 队列 |
| `mission_approvals` | 已批准方案指纹列表 |
| `target_authorizations` | 已授权目标 |
| `engagements` | 待签署武器 engagement |

### 6.2 注入 Commander Intent

```powershell
@'
{
  "id": "intent-001",
  "issued_at": 1.0,
  "issued_by": "operator",
  "objective": "优先打击 trk-1",
  "priority_tracks": ["trk-1"],
  "constraints": [],
  "roe_pref": null
}
'@ | Set-Content -Path "$env:TEMP\intent.json" -NoNewline

curl.exe -s -X POST http://127.0.0.1:4200/api/platform/intent `
  -H "Content-Type: application/json" `
  --data "@$env:TEMP\intent.json"
```

### 6.3 批准 Mission 方案（按指纹）

从 `cognitive_report.pending_approval_id` 或 Tactical 页复制指纹：

```powershell
@'
{
  "plan_id": "<fingerprint-from-pending-report>",
  "approved_by": "operator"
}
'@ | Set-Content -Path "$env:TEMP\approve-mission.json" -NoNewline

curl.exe -s -X POST http://127.0.0.1:4200/api/platform/missions/approve `
  -H "Content-Type: application/json" `
  --data "@$env:TEMP\approve-mission.json"
```

### 6.4 授权目标

```powershell
@'
{
  "platform_id": "usv-01",
  "track_id": "trk-1",
  "authorized_by": "operator"
}
'@ | Set-Content -Path "$env:TEMP\authorize-target.json" -NoNewline

curl.exe -s -X POST http://127.0.0.1:4200/api/platform/targets/authorize `
  -H "Content-Type: application/json" `
  --data "@$env:TEMP\authorize-target.json"
```

### 6.5 批准快环 Intervention（Confirm/Quorum）

快环 weapon/motion 等等待项的 `approval_id` 形如 `intervention:weapon_release:1:<intent-id>`，使用：

```powershell
@'
{
  "approval_id": "intervention:weapon_release:1:<intent-id>",
  "approved_by": "operator"
}
'@ | Set-Content -Path "$env:TEMP\approve-intervention.json" -NoNewline

curl.exe -s -X POST http://127.0.0.1:4200/api/platform/approve `
  -H "Content-Type: application/json" `
  --data "@$env:TEMP\approve-intervention.json"
```

### 6.6 签署 Engagement

```powershell
curl.exe -s -X POST "http://127.0.0.1:4200/api/platform/engagements/<engagement-id>/sign" `
  -H "Content-Type: application/json" `
  -d "{\"signer\":\"operator\",\"launch_if_ready\":true}"
```

## 8. Live 联调检查清单

按顺序执行，确认闭环可用：

- [ ] `GET /api/health` 返回 ok
- [ ] `GET /api/platform/pending` 返回新结构（含 `snapshot`、`cognitive_report`、`intent_queue`）
- [ ] `POST /api/platform/intent` 后 `intent_queue` 与 `pending_intents_count` 增加
- [ ] Confirm 模式下 `cognitive_report.pending_approval_id` 非空且 **intents 为空**
- [ ] `POST /api/platform/missions/approve` 后下一慢环周期 `intents` 非空
- [ ] `POST /api/platform/targets/authorize` 写入 `target_authorizations`
- [ ] Dashboard `#tactical` 可见五区 UI，且 `tacticalPage` 已打包进 HTML
- [ ] 离开 Tactical 页后无重复 3s 轮询（路由切换会 dispatch `page-leave`）

## 9. 常见问题

| 现象 | 可能原因 | 处理 |
|------|----------|------|
| `planning_enabled: false` | `[platform.planning] enabled = false` 或未启用 platform | 检查 config 与 primary adapter |
| `snapshot: null` | 快环未 poll 到态势 | 确认 ArkSIM/adapter 已连接并有 snapshot |
| `cognitive_report: null` | 慢环未跑或尚无 snapshot | 等待 interval；确认 planning enabled |
| Approve 后仍 Pending | 方案内容变化导致指纹变化 | 对**当前** `pending_approval_id` 重新批准 |
| intent 注入后消失但未执行 | 上一版 bug：Pending 时误消费 intent | 已修复：Pending 保留队列直至 Approved/Denied |
| `openfang.exe` 编译失败 | daemon 占用二进制 | 先 `Stop-Process` 再编译，或 `cargo build --workspace --lib` |
| PowerShell JSON 报错 | 引号转义 | 使用 `--data "@file.json"` |

## 11. 安全提示

- 勿将真实 `llm_api_key`、共享密钥提交到版本库；示例配置中的密钥请替换为你自己的值。
- Mission 审批按**方案内容指纹**生效：态势或 allocation 变化后需重新批准，避免旧批准误放行新方案。
